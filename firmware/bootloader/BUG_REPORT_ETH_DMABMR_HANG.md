# Bug Report: Ethernet DMABMR Software Reset Hangs After Bootloader Jump

**Board:** STM32F767ZI (Nucleo-F767ZI)  
**Bootloader:** OpenBLT via ChibiOS (`bootloader/`)  
**Firmware:** rusefi main app with lwIP Ethernet  
**Date:** 2026-05-22

---

## Summary

After the OpenBLT bootloader jumps to the main rusefi app, the main app's Ethernet
MAC never starts. The lwIP thread hangs indefinitely inside `mac_lld_start()` waiting
for the `ETH->DMABMR` Software Reset bit (`ETH_DMABMR_SR`) to self-clear. The board
appears to run normally (PC is in `lwip_thread`), but no ARP/IP traffic is ever
generated. Ethernet is effectively dead.

---

## Symptoms

- Board boots, main app runs (verified via OpenOCD `halt` + `reg pc`).
- PC lands in `lwip_thread.lto_priv.0` and never moves.
- `ping 192.168.10.239` fails — ICMP unreachable from gateway, no response from ECU.
- `arping 192.168.10.239` sends 5 ARP probes, receives 0 replies.
- ECU MAC address `08:00:27:69:5B:45` never appears in the ARP table.

---

## Root Cause Chain

### Step 1 — Bootloader initializes MAC driver, then powers down PHY

During `halInit()` (called from bootloader `main()`), ChibiOS calls `mac_lld_init()`
in `hal_mac_lld.c`:

```c
void mac_lld_init(void) {
  rccResetETH();          // Full hardware reset of ETH peripheral
  rccEnableETH(true);     // Enable ETH AHB clock
  // ... PHY soft-reset via MDIO ...
  mii_write(&ETHD1, MII_BMCR, BMCR_RESET);
  while (mii_read(&ETHD1, MII_BMCR) & BMCR_RESET);

#if STM32_MAC_ETH1_CHANGE_PHY_STATE   // ← DEFAULT IS TRUE
  mii_write(&ETHD1, MII_BMCR,
            mii_read(&ETHD1, MII_BMCR) | BMCR_PDOWN);  // PHY → power-down
#endif

  rccDisableETH();        // Disable ETH clock
}
```

`STM32_MAC_ETH1_CHANGE_PHY_STATE` defaults to `TRUE` in
`ChibiOS/os/hal/ports/STM32/LLD/MACv1/hal_mac_lld.h:161`.
The LAN8742A PHY on the Nucleo board is put into power-down via BMCR.PDOWN.

### Step 2 — PHY power-down stops the RMII 50 MHz reference clock

The STM32F767 ETH MAC uses RMII mode. The 50 MHz reference clock
(`ETH_REF_CLK` / `RMII_REF_CLK`) is **generated by the LAN8742A** and fed back
into the STM32. When the PHY enters power-down mode, it stops outputting this clock.

Without `RMII_REF_CLK`, the ETH MAC/DMA logic cannot execute any clock-dependent
operations — including completing a software reset.

### Step 3 — Main app's `mac_lld_start()` issues DMABMR Software Reset and hangs

The bootloader jumps to the main app (via `CpuStartUserProgram()`). The main app
runs `halInit()` → `mac_lld_init()` again (same sequence, PHY may or may not re-power
depending on state). Eventually `lwipInit()` starts the lwIP thread, which calls
`macStart()` → `mac_lld_start()`:

```c
void mac_lld_start(MACDriver *macp) {
  // ... descriptor reset ...
  rccEnableETH(true);

#if defined(STM32_MAC_DMABMR_SR)        // ← ENABLED via board.mk
  ETH->DMABMR |= ETH_DMABMR_SR;        // Set software reset bit
  while (ETH->DMABMR & ETH_DMABMR_SR)  // ← HANGS HERE FOREVER
    ;
#endif

#if STM32_MAC_ETH1_CHANGE_PHY_STATE
  // Power up PHY — but this code is NEVER REACHED
  mii_write(macp, MII_BMCR,
            mii_read(macp, MII_BMCR) & ~BMCR_PDOWN);
#endif
  // ... rest of init never executes ...
}
```

The `ETH_DMABMR_SR` bit self-clears when the DMA resets all internal logic —
but this requires the 50 MHz RMII clock to clock the DMA. Since the PHY is powered
down (no clock), the bit never clears. The thread spins forever.

### Step 4 — ETH MAC registers remain in hardware-reset state

Confirmed via OpenOCD while board is "running":

```
ETH_MACCR  @ 0x40028000: 0x00008000   (reset value — RE=0, TE=0)
ETH_DMAOMR @ 0x40029018: 0x00000000   (DMA not started)
ETH_DMABMR @ 0x40029000: 0x00020101   (bit 0 = SR still set)
```

### Disassembly evidence

The hang loop (LTO-inlined into `lwip_thread`):

```asm
0x21878e:  ldr   r3, [r5, #0]      ; r3 = ETH->DMABMR
0x218790:  ands.w sl, r3, #1       ; sl = r3 & ETH_DMABMR_SR
0x218794:  bne.n 0x21878e          ; loop while SR still set
```

## Additional Context: Earlier Bug (Fixed)

Before this hang was identified, the bootloader was HardFaulting on every boot.

**Cause:** `CpuStartUserProgram()` called `netdev_deinit()` → `macStop(&ETHD1)`
unconditionally. OpenBLT uses deferred/lazy network init — if no XCP connection
is made, `netdev_init()` / `macStart()` is never called. With
`ETHD1.state == MAC_UNINIT`, `mac_lld_stop()` tried to write to ETH peripheral
registers (MDIO, MACCR) without the ETH clock being enabled → Bus Fault → HardFault.

**Fix applied** (in `hw_layer/openblt/netdev.c`):

```c
static bool netdev_started = false;

void netdev_init(void) {
  // ... MAC address setup ...
  macStart(&ETHD1, &mac_config);
  netdev_started = true;     // ← added
  // ...
}

void netdev_deinit(void) {
  if (netdev_started) {      // ← guard added
    macStop(&ETHD1);
    netdev_started = false;
  }
}
```

This fixed the HardFault. The bootloader now jumps to the app successfully.

## Proposed Fixes

### Option A — Disable PHY power-down in bootloader (recommended)

Add to `config/boards/nucleo_f767/board.mk` (or the bootloader's compiler flags):

```makefile
DDEFS += -DSTM32_MAC_ETH1_CHANGE_PHY_STATE=FALSE
```

This prevents `mac_lld_init()` from powering down the PHY. The LAN8742A keeps
generating the 50 MHz RMII clock throughout the bootloader's lifetime, so it is
available when the main app calls `mac_lld_start()`.

**Caveat:** This means the bootloader leaves the PHY in active/auto-negotiate state
rather than power-save. Acceptable for an ECU.

### Option B — Power up PHY before DMABMR SR in `mac_lld_start()`

Reorder the ChibiOS driver so the PHY power-up step runs **before** the DMABMR
software reset:

```c
void mac_lld_start(MACDriver *macp) {
  rccEnableETH(true);

#if STM32_MAC_ETH1_CHANGE_PHY_STATE
  /* Power up PHY FIRST so RMII clock is available for DMABMR reset */
  mii_write(macp, MII_BMCR, mii_read(macp, MII_BMCR) & ~BMCR_PDOWN);
  /* small delay for clock stabilization */
#endif

#if defined(STM32_MAC_DMABMR_SR)
  ETH->DMABMR |= ETH_DMABMR_SR;
  while (ETH->DMABMR & ETH_DMABMR_SR);
#endif
  // ...
}
```

This is the correct fix for the ChibiOS driver but requires modifying a vendored file.

### Option C — Don't use DMABMR SR at all (quick workaround)

Remove `STM32_MAC_DMABMR_SR=TRUE` from `board.mk`. This skips the software reset
of the DMA. If the bootloader never started the ETH DMA (deferred init, no XCP
connection), there is nothing to reset — the flag is only strictly needed when
the bootloader leaves DMA running, which the `netdev_deinit()` fix already handles.
